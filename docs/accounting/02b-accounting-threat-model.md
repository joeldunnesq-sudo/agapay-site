# AGAPAY Accounting — Threat Model

Supporting document to `docs/accounting/02-phase-0.75-foundational-readiness.md` (Workstream 12). Severity: Critical / High / Medium / Low.

| # | Threat | Asset | Actor | Attack path | Likelihood | Impact | Existing mitigation | Missing mitigation | Required control | Verification test | Resolve by |
|---|---|---|---|---|---|---|---|---|---|---|---|
| 1 | Parish A accesses Parish B's accounting data | Parish B's accounting D1 | A parish-level actor (compromised or malicious) | Reused/forged session against wrong parish context; a route that trusts a client-supplied parish ID | Medium (no capability model exists yet to prevent it) | Critical | None specific to accounting yet — Phase 0 confirmed today's model (shared bearer per parish) provides only application-layer scoping | Server-side, capability-checked, registry-resolved parish authorization on every accounting call | Accounting Gateway resolves parish context server-side only (Workstream 3/5) | Automated cross-parish denial test (Workstream 8) | Before Phase 1 control-plane goes live |
| 2 | Forged parish IDs | Any parish-scoped record | External or internal attacker | Manipulating a client-supplied `parish_id` field/parameter | Medium | Critical | Existing non-accounting routes already resolve parish context via server-side registration lookup in most paths (Phase 0 finding) — a reasonable pattern to carry forward, but not yet formalized as a rule for the accounting domain specifically | Explicit rule: accounting domain never accepts a parish identifier as authoritative from the client | Same as #1 | Same as #1 | Before Phase 1 |
| 3 | Forged database binding identifiers | Accounting Gateway's binding resolution | Internal (compromised main-Worker code or a malicious request crafted to look like an internal call) | Attempting to pass a binding name/identifier through to the gateway from outside its trust boundary | Low (Service Bindings are not publicly reachable, confirmed per `02e`) | Critical | Service Binding architecture itself (not publicly routable) — confirmed via Cloudflare documentation | Gateway must still never accept a binding name as a parameter from its caller — it must derive the binding itself from the registry, using only a parish identity already authorized upstream | Gateway performs its own registry lookup; never receives "which binding to use" as an argument | Test that the gateway's public RPC surface has no method accepting a raw binding/database identifier | Before Phase 1 |
| 4 | Stolen shared parish bearer token | Any parish-dashboard-gated route (non-accounting, ongoing) and, if accounting mistakenly reused it, accounting data | External attacker with a leaked token | Token exposure (logging, phishing, interception) | Medium (a single shared secret is inherently a bigger blast radius than per-person credentials) | High for non-accounting features; **must be Critical-equivalent-prevented for accounting** by architectural exclusion | None beyond the token itself being a bearer secret | Rotation capability already exists (`parishDashboardToken` can be regenerated) for non-accounting use; accounting's mitigation is architectural exclusion, not token hardening | Accounting routes reject shared bearer tokens outright, by design (Workstream 3) | Test: a valid shared bearer token, with no platform-user session, is rejected by every accounting route | Before Phase 1 |
| 5 | Compromised parish user | Parish's own accounting data (within their authorized scope) | External attacker with a legitimate parish user's stolen credentials | Credential theft (phishing, reuse, etc.) | Medium | High — bounded to that parish's data, which limits blast radius but is still serious for that parish | None yet (no individual parish-user identity exists today) | Session expiry, capability-scoping (limits what a compromised low-privilege user can do even within their own parish), reauthentication for high-risk actions | Reauthentication requirement for check issuance / period reopening (Workstream 3 open decision) | Manual security review + reauthentication test | Before pilot |
| 6 | Excessive role assignment ("everyone is treasurer") | Separation-of-duties integrity | A well-meaning but careless parish admin | Granting broad capabilities by default rather than minimum necessary | High likelihood for small missions (legitimately, per Philosophy §23) | Medium — acceptable for genuinely small missions, a real control gap for larger parishes that should be separating duties but aren't | None yet | Visibility into duty-combination (Philosophy §23: "the system must reveal when one person performs multiple sensitive actions") | A report/flag surfacing same-actor combinations on the same transaction | Test that combined-duty actions are flagged in an available report | Before pilot for Parish-tier; not required for Mission-tier |
| 7 | Privilege escalation | Capability model integrity | A parish user with limited capabilities | Exploiting a bug that fails to check a capability, or a role-template expansion bug that grants more than intended | Low if capability checks are centralized and tested; higher if checks are duplicated ad hoc (an anti-pattern this program explicitly prohibits, Philosophy §29 item 13) | Critical | None yet (no capability model exists) | Centralized, single-implementation capability enforcement (Workstream 3/Philosophy §5) | Capability-boundary tests per Workstream 3/8 | Before Phase 1 |
| 8 | AGAPAY support misuse | Any/all parish accounting data | An AGAPAY staff member (insider risk) | Using support access beyond its intended scope, or without a documented reason | Low likelihood, high consequence given breadth of potential access | Critical | Existing central `audit_log` table (Phase 0 finding) provides a real foundation for logging *an* admin action, though not yet a support-specific, time-limited access grant | Exceptional, reasoned, time-limited (where practical) support-access grant, logged both centrally and parish-locally (Philosophy §22/§23, invariant 15) | Dedicated support-access workflow, distinct from ordinary admin auth | Test that support access requires an explicit grant + reason, is logged in both audit surfaces, and (where implemented) expires | Before pilot |
| 9 | Cloudflare API token compromise | Any resource the token can reach | External attacker | Token leaked via logs, source control, or a compromised CI secret | Low likelihood, high blast radius depending on token scope | High to Critical depending on scope | `CLOUDFLARE_API_TOKEN` already exists as a GitHub Actions secret (confirmed in `.github/workflows/deploy.yml`) and is not visible in this repository's source (confirmed no plaintext secrets found) | Scoped, least-privilege token permissions (not independently verified in this pass — token scope is a Cloudflare-dashboard fact, not a repo fact); avoiding Option C (REST-API D1 access) for the primary ledger path specifically reduces how often a D1-capable token needs to exist in a live Worker at all | Confirm CI token scope; prefer binding-based D1 access (Option B) over token-based access wherever possible | Manual review of token permissions in Cloudflare dashboard | Before Phase 1 (verification), before pilot (any scope tightening) |
| 10 | Webhook replay | Financial posting integrity | External attacker replaying a captured, validly-signed webhook payload | Stripe's signature includes a timestamp checked for staleness (confirmed: `verifyStripeWebhook`, `src/handlers/stripe.js:142–162`, rejects payloads more than 300 seconds old) — this substantially mitigates replay | Low, given the existing timestamp check | Medium (bounded by the 300-second window) | **Confirmed present and effective**: timestamp + signature verification | None material found | — | Existing behavior; recommend an explicit regression test asserting the 300-second bound isn't accidentally loosened | Already adequate; monitor for regression |
| 11 | Queue-message forgery | Future background-job integrity | Internal (a bug) or external (if a queue were ever exposed) | A forged or malformed message reaching a future accounting queue consumer | N/A today (no queues exist yet, confirmed) | High, once background processing exists | N/A | Message-envelope validation, tenant-authorization on every job (Workstream 6 requirement) | Design requirement for whatever background primitive is chosen | Test once background infrastructure exists | Before any queue/workflow is built (Workstream 6) |
| 12 | Migration-file tampering | Aplos migration integrity | Internal (compromised staging/storage) or external (if migration files were ever publicly accessible) | Altering a staged import file between upload and processing | Low if storage is private and access-controlled | High (could corrupt opening balances for a whole parish) | Existing private-R2 pattern (`TAX_EXEMPTION_DOCS`, `GIVING_STATEMENTS` — no public URL, ever) is a good model to extend | File hashes/checksums for migration files specifically (Philosophy §25) | Checksum verification at import time | Test that a tampered file fails validation | Before Aplos migration feature ships (beyond Phase 0.75 scope) |
| 13 | Check-PDF exposure | Bank account/routing details on check images | External attacker or unauthorized internal actor | Guessing or obtaining an R2 object key/URL for a check PDF | Low if the existing private-bucket pattern (no public `r2.dev` URL, authenticated streaming only) is followed for the new accounting bucket | Critical if it happens (real bank details) | Existing pattern for `TAX_EXEMPTION_DOCS`/`GIVING_STATEMENTS` is directly reusable and already avoids public URLs | Explicit "heightened protection" requirement for checks specifically (Philosophy §25) — not yet implemented since checks don't exist yet | Follow the existing private-document pattern exactly; consider access logging specifically for check PDFs given their sensitivity | Test that check PDF URLs are never public and require authenticated, parish-scoped access | Before check printing ships (beyond Phase 0.75 scope, but bucket design should anticipate it now — Workstream 10) |
| 14 | R2 object-key guessing | Any private accounting document | External attacker | Enumerating/guessing opaque R2 keys | Low — confirmed existing pattern uses cryptographically random keys (`generateStorageKey()`, 32 random bytes, `src/lib/tax-exemption-storage.js:62–67`) | Medium (mitigated by key entropy, but access is enforced by app-layer authorization, not key secrecy alone — Phase 0 finding) | Random key generation confirmed present | Continue app-layer authorization as the actual control, not key secrecy | Explicit authorization check before every stream-out, regardless of key entropy | Existing pattern; extend to new accounting bucket (Workstream 10) | Ongoing discipline, verify for new bucket before it's created |
| 15 | Backup exposure | Full parish accounting history in one artifact | External or internal actor | Unauthorized access to a backup export file | Low if backups follow the same private-bucket discipline | Critical (a backup is the highest-value single target — full history in one place) | Existing `docs/BACKUP_RESTORE_RUNBOOK.md` discipline for the central database is a reasonable foundation to extend | Per-parish backup access control, restore-authorization requirement (Philosophy §27) | Backups stored with the same private-document discipline; restore requires explicit, audited authorization | Test backup file access requires authorization | Before any parish accounting database is provisioned |
| 16 | Staging-to-production crossover | Both environments | Internal (configuration error) | A staging deployment accidentally pointed at a production binding, or vice versa | Medium — **confirmed real risk today**, because AGAPAY currently has no formally declared staging environment in `wrangler.toml` (Phase 0 finding, reconfirmed) | Critical if it happens (could write test data into production accounting records, or expose production data to a less-controlled staging surface) | None yet | Formal environment separation (Workstream 7) — distinct bindings, distinct secrets, no shared production binding reachable from a staging deploy | Environment-isolation test verifying a staging deploy cannot reach a production D1 binding | Before any accounting migration is tested against anything but a fully isolated environment |
| 17 | Logging of sensitive data | Bank details, tax IDs, full check data, invoice contents | Internal (a logging bug) or anyone with log access | Structured logs inadvertently including a sensitive field | Medium — plausible without an explicit redaction rule, since existing structured logging (`src/lib/logging.js`) was not built with accounting-specific sensitive fields in mind | High | Existing structured logging/audit infrastructure is a reasonable foundation | Explicit redaction rule set for accounting-specific sensitive fields (Workstream 9) | Redaction rules + a test asserting sensitive fields never appear in log output | Before any accounting feature that touches bank details/tax IDs/check data ships |
| 18 | SQL injection | Any D1-backed accounting table | External attacker via crafted input | Unparameterized SQL construction | Low — **confirmed low risk today**: every D1 call sampled and read in this and the prior audit uses parameterized `.prepare().bind()` calls, no string-concatenated SQL found | Critical if it ever occurred | Confirmed parameterized-query discipline throughout `d1First`/`d1All`/`d1Run`/`d1Batch` usage | Maintain this discipline explicitly as a rule for accounting-domain code (Philosophy doesn't currently state this as its own invariant — worth adding as an implementation-standard note in Phase 1 design, though not itself an accounting-policy invariant) | Code review discipline; consider a lint rule if one becomes practical | Ongoing | N/A — already a strength to preserve |
| 19 | Broken authorization in service-to-service calls | Accounting Gateway's RPC surface | Internal (a bug in the main Worker, or a compromised main-Worker request path) | The main Worker calling the gateway with a capability check that was itself bypassed or wrong | Medium until the capability model is built and tested | Critical | N/A yet | Capability check happens *before* the gateway is called, and the gateway performs its own minimal validation rather than blindly trusting the caller (defense in depth) | Test: gateway RPC methods reject calls missing required authorization context, even if the main Worker's own check were somehow bypassed | Before Phase 1 |
| 20 | Database provisioning race conditions | Registry integrity — "one active accounting database per parish" (Workstream 5 requirement) | Internal (two near-simultaneous provisioning requests for the same parish) | Concurrent requests both passing a "not yet provisioned" check before either completes | Low but plausible without an explicit guard | High (could result in two live accounting databases for one parish, an unrecoverable ambiguity about which is authoritative) | N/A yet | Idempotent, uniquely-constrained provisioning (Workstream 5 requirement) | Concurrency test: two simultaneous provisioning requests for the same parish produce exactly one database | Before Phase 1 |
| 21 | Database deletion or orphaning | Availability of parish accounting data | Internal (a bug or an over-broad admin tool) | Accidental or unauthorized deletion, or a provisioning failure leaving a database bound but unregistered (or registered but unbound) | Low likelihood, catastrophic impact | Critical | N/A yet | "Database deletion never occurs casually or automatically" (Workstream 5 requirement); registry and binding state must never diverge silently | Test: registry state and actual bound-database state are checked for consistency; deletion requires an explicit, separately-authorized action | Before Phase 1 |
| 22 | Failed migration leaving a tenant in an unknown state | Data integrity for a specific parish | Internal (a partial migration failure) | Fan-out migration run interrupted partway through the set of parish databases | Medium once many parishes exist | High | N/A yet (no fan-out orchestration exists) | Explicit failure-quarantine state in the registry state machine (Workstream 5/11) — a parish whose migration failed is never silently treated as current | Migration-failure test: an interrupted fan-out leaves affected parishes in a clearly-flagged, non-`active` state | Before Phase 1 |
| 23 | Replay of provisioning requests | Registry integrity | External or internal (a retried request) | A retried provisioning call after a prior one already succeeded | Low if provisioning is idempotent (a stated Workstream 5 requirement) | Medium | N/A yet | Idempotent provisioning keyed by parish identity | Test: replaying a provisioning request after success is a safe no-op, not a second database | Before Phase 1 |
| 24 | Multiple active accounting databases for one parish | Source-of-truth integrity | Internal (a registry bug or a race condition, overlapping with #20) | Any path that allows two `active`-status registry rows for the same parish | Low if uniquely constrained | Critical (breaks "authoritative accounting record" itself — Accounting Philosophy §4 has no meaning if there are two competing "authoritative" databases) | N/A yet | A uniqueness constraint (one active accounting database per accounting entity, per Workstream 5's stated required rule) | Test: attempting to activate a second database for an already-active parish is rejected | Before Phase 1 |

## Top ten risks (ranked by combined likelihood × impact × how directly they undermine the Accounting Philosophy's core invariants)

1. **#1/#2 — Cross-parish access via unresolved server-side authorization** (the single most direct violation of Philosophy invariant 15 possible).
2. **#7 — Privilege escalation from a non-centralized capability model.**
3. **#24 — Multiple active accounting databases for one parish** (breaks the entire source-of-truth concept).
4. **#8 — AGAPAY support misuse without a dedicated, audited access path.**
5. **#16 — Staging-to-production crossover**, confirmed as a real current gap (no formal environments exist).
6. **#20/#21/#22/#23 — Registry/provisioning integrity failures**, several distinct but related risks around the control-plane registry (Workstream 5) being the thing everything else depends on.
7. **#4 — Continued reliance on the shared parish bearer token for anything accounting-related**, if the architectural exclusion in Workstream 3 isn't followed rigorously.
8. **#9 — Cloudflare API token scope**, unverified in this pass and a real unknown.
9. **#17 — Sensitive-data logging**, plausible without an explicit rule and easy to get wrong by default.
10. **#15 — Backup exposure**, high-value single target once real accounting data exists.

## Phase 1 blockers (must be resolved before Phase 1 control-plane work begins)

#1, #2, #3, #7, #19, #20, #21, #22, #23, #24 — all of the identity/authorization and registry-integrity risks. These are not optional hardening; they are the actual subject matter of Phase 1.

## Pilot blockers (must be resolved before any real parish's data is at stake)

#5, #6 (visibility, not prevention, for Mission-tier), #8, #9 (verification), #13, #14 (verification for the new bucket), #15, #16, #17.

## General-release blockers (can follow pilot, but must precede scaling beyond a handful of parishes)

#11, #12 (once migration/background processing exist), #18 (ongoing discipline, formalize as a standard).
