# Stripe Accounting-Readiness Matrix

Supporting document to `docs/accounting/02-phase-0.75-foundational-readiness.md` (Workstream 2). This reflects a full, line-by-line read of `src/handlers/stripe.js` (900 lines, all of it — not a sample) plus the fee-capture logic in `src/handlers/parish.js` (`stripePaymentIntentFinancialUpdates`, lines 1452–1511) and the commerce-order Stripe reconciliation functions (`completeCommerceOrderFromStripe`, `refundCommerceOrderFromStripe`, `src/handlers/parish.js:5503–5585`). Idempotency mechanics are from `claimStripeEvent`/`finishStripeEvent` in `src/lib/core.js:666–751`.

## Three distinct kinds of idempotency (do not conflate)

1. **Platform webhook idempotency** — "has this exact Stripe event ID already been claimed/processed." Enforced today by the `stripe_events` table (`INSERT ... ON CONFLICT(id) DO NOTHING`), with stale-processing and failed-event retry logic. **Confirmed present and well-built.**
2. **Operational-record idempotency** — "does processing this event twice produce the same `donor_offerings`/`commerce_orders` row state, not a duplicate row." Confirmed present in most handlers via upsert-by-checkout-session-id / by-payment-intent-id lookups, and explicit re-entrancy guards (e.g. `completeCommerceOrderFromStripe` returns early if `order.payment_status === "paid"`). Not independently tested per-event-type in this pass — flagged for the accounting-specific test suite in Workstream 8.
3. **Future accounting-posting idempotency** — does not exist yet, because the posting engine does not exist yet. This must be built new in Phase 1, using each event's Stripe event ID (or a derived key) as the posting engine's idempotency key per Accounting Philosophy §10. **Not to be assumed satisfied by #1.**

## Corrected Phase 0 finding

Phase 0 flagged Stripe fee capture and refund/dispute handling as **unconfirmed** because `processStripeWebhookEvent` was sampled, not fully read. Having now read the full file: **fees, gross, and net amounts ARE captured**, sourced from Stripe's own Balance Transaction object (not estimated) whenever available — `stripePaymentIntentFinancialUpdates` expands `latest_charge.balance_transaction` and records `stripeFeeCents`, `agapayFeeCents`, `totalFeeCents`, `parishNetCents`, `stripeBalanceTransactionId`, and a `stripeFeeSource` flag distinguishing `"balance_transaction"` (real) from `"estimated"` (fallback). **Refunds and disputes are also handled** (`charge.refunded`, `charge.dispute.created`, `charge.dispute.closed`). **Payouts are not handled at all** — no `payout.*` event type appears anywhere in the codebase (confirmed by exhaustive grep, zero matches).

## Cross-cutting ordering note

Stripe does not guarantee webhook delivery order, especially under retries. This codebase's current handling is generally tolerant of that because most handlers perform idempotent `UPDATE`s keyed by a stable identifier (checkout session ID or payment intent ID) rather than depending on receiving events in a specific sequence — **except** for the dispute lifecycle, where `charge.dispute.closed` updating a `donor_offerings` row assumes a prior `charge.dispute.created` already set `stripeDisputeId` context; if `closed` somehow arrived without a prior `created` being processed, the row would still update correctly (the `UPDATE` doesn't require the prior state), but this has not been explicitly tested. Recommended for the accounting-specific test suite (Workstream 8): explicit out-of-order and duplicate-delivery tests for the dispute pair and for the `checkout.session.completed` / `payment_intent.succeeded` pair.

## Summary of gaps requiring remediation before ledger development

1. **Dual posting-trigger risk** (`checkout.session.completed` vs. `payment_intent.succeeded` for the same payment) — must pick one canonical posting trigger.
2. **Commerce refund amount not persisted as an exact figure** — needed for accurate contra-revenue posting.
3. **Commerce disputes not reflected on `commerce_orders` at all** — a real data gap, newly confirmed by this full read.
4. **No payout event handling anywhere** — blocks automated payout-side posting only, not donation/fee-side posting.
5. **Stripe Connect webhook subscription list** — needs confirmation against the Stripe dashboard, not just the code, since the code is built to accept Connect events but this repo can't confirm which event types Stripe is actually configured to send.

None of these block Phase 0.75 control-plane work. Items 1–3 must be resolved before ledger/posting-engine development (Phase 1+); item 4 blocks automated payout posting specifically and can be deferred if payout reconciliation starts as a manual/reported process; item 5 is a verification task, not a code change.
